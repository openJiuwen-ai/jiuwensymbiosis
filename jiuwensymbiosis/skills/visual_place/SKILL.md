# visual_place — 视觉/坐标引导放置（夹爪 / 吸盘通用）

## 何时启用

满足以下**全部**条件时启用本 skill：

1. 用户指令含"把（手上的 / 刚抓的）X 放到 Y / 放下 / 放进去"等放置意图。
2. 机器人 `api.capabilities` 包含 `motion.cartesian`，且包含**至少一种抓取能力**：`grasp.parallel` **或** `grasp.suction`；放置点由像素定位时还要 `vision.detection`。
3. 已注册 `robot_control` 工具。
4. **前置条件：夹具已经抓住目标物体**。本 skill 假设刚刚成功跑完 `visual_pick`（或上层任务已显式抓住物体），TCP 位于 pick 完成时的 `(x, y, z + lift)` 搬运高度。
5. 已经知道放置点 `(px, py, pz)`：要么由用户/上层显式给出 mm 坐标，要么需要本 skill 通过 `get_grasp_info_simple` / `analyze_scene` 在线定位。

如果 4 不满足，**不要**启用本 skill —— 空跑会让夹具在错误位置释放、并可能把空气当成物体释放。

## 抓取 / 释放动作（按机器人能力选择）

与 visual_pick 同一张映射表。本 skill 只用到**释放**：

| 你的能力 | 抓取 action（本 skill 不调） | 释放 action |
|---|---|---|
| `grasp.parallel` | `close_gripper` | `open_gripper` |
| `grasp.suction` | `activate_suction` | `deactivate_suction` |

下文用 **`<释放>`** 指代上表里你机器人对应的释放动作。

## 放置目标来自用户任务（不要用户传参数）

放置目标的名称**由用户的自然语言任务决定**：从用户这句话里识别"放到哪里"，用它的自然语言描述
（颜色/形状/大小/类别/材质/位置等任意特征的组合，也可能只有类别）作为检测目标。用户只说任务、
不会再单独传参数；**放置目标一律以用户任务为准，不要套用本文出现过的任何示例物体名**。

- 若 visual_pick 阶段已在 home 处把放置目标坐标一并读好（腕部相机推荐做法），**直接用那组坐标**，不要再检测。

### 放置高度直接用检测给的 `place_z`，不要自己算

`get_grasp_info_simple`（放置目标那次）返回里已经有**确定化算好**的放置高度：

```
{ "position": [px, py, top_z],  "place_z": <数>,  "place_position": [px, py, place_z], ... }
```

- 放置点 xy = 放置目标的 `position[0], position[1]`（记作 `px, py`）。
- **放置高度 = 放置目标的 `place_z`**（= 放置面高度 + 已配置的堆叠偏移，已算好）。降到 `place_z` 松爪，被抓物底面正好落在放置面上，既不压进去也不悬空。
- **不要**自己拿 `position` 的 z 去加减算放置高度。

## 标准 Workflow

### 分支 A：放置点坐标已知（首选）

`px, py` 为放置目标 xy；`place_z` 用放置目标检测返回的 `place_z`。

| # | action            | params                                          | 目的                                            |
|---|-------------------|-------------------------------------------------|-------------------------------------------------|
| 1 | `goto_xyzr`       | `{"x": px, "y": py, "z": place_z + lift}`      | 横移到放置点正上方（lift ≈ +50~80mm）。         |
| 2 | `goto_xyzr`       | `{"x": px, "y": py, "z": place_z}`             | 下降到放置高度 `place_z`（检测返回的，别改）。  |
| 3 | `<释放>`          | `{}`                                            | 释放物体（夹爪张开 / 吸盘关闭）。               |
| 4 | `goto_xyzr`       | `{"x": px, "y": py, "z": place_z + lift}`      | 提起到搬运高度，避免拖到刚放下的物体。          |
| 5 | `home`            | `{}`                                            | 回到 home 位姿，任务结束。                      |

### 分支 B：放置点需要先视觉定位

只在用户只描述了语义放置点（例如"放到红色框子里"）而没给坐标时使用。

| # | action            | params                                       | 目的                                            |
|---|-------------------|----------------------------------------------|-------------------------------------------------|
| 0a| `get_grasp_info_simple` | `{"object_name": "<放置点语义>"}`     | 检测放置容器/区域，返回 `{ok, position:[px,py,pz], ...}`。 |

> ⚠️ 分支 B 的 0a **不能 home 重拍**：home 会让机械臂回到拍照位姿，但此时夹具上挂着物体，回 home 可能拖动其它东西、或让物体进入视场遮挡放置点。**继续在当前搬运高度拍照**，接受深度噪声略大；如果 `score < 0.4`，**直接退回失败处理**，让上层用坐标分支重来。

跑完 0a 后回到分支 A 的 1~5。

## 结束状态

夹具空载、物体在 `(px, py, pz)`、TCP 在 home。

## 失败处理

放置过程中失败的代价比抓取期失败更大（物体已悬空），按如下优先级处理：

1. **如果在分支 B 的 0a 失败**（识别置信度低 / 投影报错）：物体仍在夹具上，**不要**强行释放。简短报告"放置点未识别"，把控制权交还上层 agent —— 上层可改用坐标分支重新调用本 skill，或回退到 visual_pick 的 home 流程清场。
2. **如果在分支 A 步骤 1~2 失败**（goto 越界/碰撞）：**不要**在原地 `<释放>`（会把物体丢在错误位置）。优先尝试 `goto_xyzr` 抬回搬运高度 `(current_x, current_y, current_z + lift)`，再 `home`，再上报。
3. **如果在分支 A 步骤 3 失败**（`<释放>` 报错）：物体可能已经在放置点上、可能还没。优先 `goto_xyzr` 抬到 `(px, py, pz + lift)` 拉开距离再 `home`，**不要**重复 `<释放>`，重试有把第二个物体扔到同位置的风险（如果第一次其实成功了）。
4. **如果在分支 A 步骤 4~5 失败**（提起/home）：尝试再调一次 `home`；仍失败就上报"放置成功但回 home 失败，需手动复位"。

## 与 Rails 的协作

- **SafetyRail**：放置点 z 通常比 pick 点低，必须先把放置容器的高度计算正确，否则会被 SafetyRail 以越过 z_min_safe 拦下。
- **VisualFeedbackRail**：分支 B 的视觉定位会触发反馈；本 skill **预期**视觉噪声更大，不要轻信反馈给的位置纠偏。
- **RecoveryRail**：会兜底 home + release。本 skill 的失败处理之所以**克制 `<释放>`**，就是为了不与 RecoveryRail 的 release 形成双重释放（导致重复事件 / 状态混乱）。

## 参数取值约定

- `lift`（搬运/抬起高度偏移）= 50 ~ 80 mm，与 visual_pick 保持一致。
- 所有坐标 mm / deg；放置点 z 是物体放置面的高度，**不是**夹具到物体顶面的距离。

## Anti-patterns（不要做）

- ❌ 在没有"已抓住物体"前置条件时启用本 skill：会让夹具在空气中 `<释放>`，下游任务以为放置完成。
- ❌ 在本 skill 内部调抓取动作（`close_gripper` / `activate_suction`）：抓取是 visual_pick 的职责。
- ❌ 失败后立即 `<释放>`：可能让物体掉在错误位置。先抬到 lift 高度再说。
- ❌ 调用不在 `api_capabilities` 内的动作：会返回 unknown action。
- ❌ 用 `run_python`/`robot_control` 之外的工具实现本 workflow：会绕过 SafetyRail。
